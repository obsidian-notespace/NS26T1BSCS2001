5. Functional Dependencies (FDs)

Definition

A functional dependency is a constraint between two sets of attributes in a relation. It is denoted as \( \alpha \rightarrow \beta \), where \( \alpha \) and \( \beta \) are subsets of the relation’s attributes. The constraint means: for any two tuples \( t_1 \) and \( t_2 \) in any valid instance of the relation, if \( t_1[\alpha] = t_2[\alpha] \), then \( t_1[\beta] = t_2[\beta] \). In other words, the value of \( \alpha \) uniquely determines the value of \( \beta \).

Explanation

FDs are a formal way to capture real‑world business rules. For example, in a university database, the rule “each department has a unique building” is expressed as \( \text{dept\_name} \rightarrow \text{building} \). This must hold for all possible data, not just the current instance.

FDs generalize the concept of keys: a superkey \( K \) satisfies \( K \rightarrow \text{all attributes} \). A candidate key is a minimal superkey.

It is important to note that FDs are not derived from data; they are derived from the semantics (meaning) of the real world. Even if a given instance accidentally satisfies \( \text{name} \rightarrow \text{ID} \) (e.g., all names are unique), we cannot conclude it is a valid FD because two people can have the same name in reality.

FDs are used in normalization to identify redundancy and guide decomposition.

Intuition / Mental Model

Think of an FD as a mathematical function: if you know the input (\( \alpha \)), you can compute the output (\( \beta \)) deterministically. For instance, knowing a person’s Social Security number (SSN) determines their name, address, etc. That’s \( \text{SSN} \rightarrow \text{name} \).

Key Definitions

  • Functional Dependency (FD): \( \alpha \rightarrow \beta \) means \( \alpha \) uniquely determines \( \beta \).
  • Trivial FD: An FD where \( \beta \subseteq \alpha \) (e.g., \( AB \rightarrow A \)). Always true.
  • Non‑trivial FD: \( \beta \not\subseteq \alpha \) (e.g., \( A \rightarrow B \)).
  • Closure of FD set (\( F^+ \)): All FDs that can be derived from \( F \) using Armstrong’s axioms.

Formulas / Examples

Example 1: In the relation instructor(ID, name, dept_name, salary), we may have \( ID \rightarrow name, dept\_name, salary \) because ID is a candidate key. Also, \( dept\_name \rightarrow building \) is an FD if each department has a single building (but building is not in this relation; it would be in a separate department relation).

Example 2: Consider the instance:

A B
1 4
1 5
2 6

Here \( A \rightarrow B \) does NOT hold because A=1 gives two different B values (4 and 5). However, \( B \rightarrow A \) might hold if 4→1,5→1,6→2 (which it does in this instance). But we cannot declare it as a true FD unless it holds for all possible future instances.

Common Mistakes

  • Misconception: “If an FD holds in the current data, it is a valid FD.” (Correction: FDs are semantic rules that must hold for all possible instances, not just the current snapshot.)
  • Misconception: “FDs are the same as keys.” (Correction: Keys are a special case where the right side is all attributes, but FDs can be any subset.)
  • Misconception: “Trivial FDs are useless.” (Correction: They are important in inference algorithms and as base cases.)

Edge Cases

  • An FD with an empty left side: \( \emptyset \rightarrow \beta \) means that all tuples have the same value for \( \beta \) (a constant).
  • An FD with an empty right side is trivial and meaningless; not used.

Real‑world Application

FDs are used by database designers to enforce business rules. For example, in a flight booking system, \( \text{flight\_number} \rightarrow \text{airline} \) ensures that a flight number always belongs to the same airline.

Key Takeaways

• FDs are semantic constraints, not data‑derived.
• \( \alpha \rightarrow \beta \) means α uniquely determines β.
• Trivial FDs have β ⊆ α.
• FDs generalize the concept of keys.
• They are the foundation of normalization theory.

6. Armstrong’s Axioms

Definition

Armstrong’s axioms are a set of three inference rules (reflexivity, augmentation, transitivity) that are sound and complete for deriving all functional dependencies implied by a given set \( F \).

Explanation

Given a set of FDs \( F \), we often need to know what other FDs must also hold. Armstrong’s axioms provide a mechanical way to derive them.

  • Reflexivity: If \( \beta \subseteq \alpha \), then \( \alpha \rightarrow \beta \). (Trivial dependencies.)
  • Augmentation: If \( \alpha \rightarrow \beta \), then \( \alpha\gamma \rightarrow \beta\gamma \) for any set of attributes \( \gamma \). (Adding the same attributes to both sides preserves the dependency.)
  • Transitivity: If \( \alpha \rightarrow \beta \) and \( \beta \rightarrow \gamma \), then \( \alpha \rightarrow \gamma \).

Soundness means that any FD derived using these axioms is indeed implied by \( F \). Completeness means that any FD implied by \( F \) can be derived using these axioms. Therefore, \( F^+ \) (the closure of F) is exactly the set of FDs obtainable by repeatedly applying these three rules.

From these three, we can derive additional rules such as Union, Decomposition, and Pseudo‑transitivity, which are convenient shortcuts.

Intuition / Mental Model

Think of Armstrong’s axioms as the algebra of functional dependencies. Just as you can manipulate equations (e.g., if a=b and b=c then a=c), you can manipulate FDs. Reflexivity says “obvious” dependencies (like \( AB \rightarrow A \)) are always true. Augmentation says you can add the same “junk” to both sides. Transitivity chains dependencies.

Key Definitions

  • Sound: Every derived FD is valid.
  • Complete: Every valid FD can be derived.
  • Closure of F (\( F^+ \)): The set of all FDs derivable from F using Armstrong’s axioms.

Formulas / Examples

Example: Given \( F = \{ A \rightarrow B, B \rightarrow C \} \).

  • By transitivity: \( A \rightarrow C \).
  • By augmentation: \( A \rightarrow B \) gives \( A A \rightarrow A B \) i.e., \( A \rightarrow AB \) (reflexivity also applies).
  • Also \( B \rightarrow C \) augmented with \( D \): \( BD \rightarrow CD \).

Thus \( F^+ \) includes at least \( A \rightarrow C \) and all trivial FDs.

Derived Rules

  • Union: If \( \alpha \rightarrow \beta \) and \( \alpha \rightarrow \gamma \), then \( \alpha \rightarrow \beta\gamma \).
  • Decomposition: If \( \alpha \rightarrow \beta\gamma \), then \( \alpha \rightarrow \beta \) and \( \alpha \rightarrow \gamma \).
  • Pseudo‑transitivity: If \( \alpha \rightarrow \beta \) and \( \beta\gamma \rightarrow \delta \), then \( \alpha\gamma \rightarrow \delta \).

Common Mistakes

  • Misconception: “Augmentation can only be used with new attributes.” (Correction: \( \gamma \) can be any set, including attributes already present.)
  • Misconception: “Transitivity requires the middle set to be exactly the same.” (Correction: It requires \( \beta \rightarrow \gamma \), not necessarily \( \beta \) equals the right side of the first FD exactly; subsets are fine.)
  • Misconception: “Derived rules are axioms.” (Correction: They are theorems that follow from the three axioms.)

Edge Cases

  • If \( F \) is empty, \( F^+ \) contains only trivial FDs (reflexivity).
  • Applying augmentation with \( \gamma \) that is already in \( \alpha \) or \( \beta \) may produce redundant but still valid FDs.

Real‑world Application

Database design tools use Armstrong’s axioms to compute the closure of a set of FDs, which is used to find candidate keys and check normal forms.

Key Takeaways

• Three axioms: Reflexivity, Augmentation, Transitivity.
• Sound and complete: they derive all implied FDs and nothing else.
• Derived rules (Union, Decomposition, Pseudo‑transitivity) are useful shortcuts.
• The closure \( F^+ \) is the set of all FDs that hold.
• These axioms form the basis for algorithmic normalization.
