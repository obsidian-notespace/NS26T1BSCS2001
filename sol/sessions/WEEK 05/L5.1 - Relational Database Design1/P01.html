1. Goals of a Good Relational Database Design

Definition

A good relational database design is one that accurately reflects the real‑world structure of the data, avoids unnecessary duplication, ensures efficient access, preserves data integrity, and is easy to understand and maintain over the lifetime of the database.

Explanation

Designing a relational database is not just about storing data; it is about creating a blueprint that faithfully captures the semantics of the application domain. The following design goals, as emphasised in the lecture, are the cornerstone of professional database development:

  • Real‑world fidelity: The schema must mirror the actual entities, attributes and relationships that exist in the problem domain. Any mismatch leads to incorrect query results and user confusion.
  • Future‑proofness: A good design should be able to accommodate all data that will ever be stored, even if the exact values are not known at design time. The structure should not artificially prevent legal data from being inserted.
  • Redundancy avoidance: Storing the same fact in multiple places wastes storage and, more importantly, introduces the risk of inconsistency. Redundancy is the enemy of integrity.
  • Access efficiency: Queries must run in acceptable time. While normalisation reduces redundancy, sometimes a small amount of controlled redundancy is introduced for performance (denormalisation) – but that is a deliberate trade‑off, not a design goal per se.
  • Integrity support: The schema should enable the enforcement of all business rules (e.g., via keys, foreign keys, check constraints). Functional dependencies are a way to express such rules formally.
  • Clean, consistent and understandable: Other developers (and your future self) must be able to grasp the design quickly. A chaotic, over‑complicated schema leads to costly errors.

The lecture stresses that some of these goals conflict. For example, eliminating redundancy often requires splitting tables, which can slow down queries that need to re‑join them. The art of database design is to find the right balance, typically guided by normalisation theory.

Intuition / Mental Model

Think of a well‑designed database as a neatly organised filing cabinet. Each folder (table) holds only one kind of document (entity). Documents never repeat the same information – if a piece of information appears in two folders, updating it in one but forgetting the other creates chaos. A good design ensures that every fact is stored exactly once, and every folder can be opened without needing to know about other folders (insertion anomaly prevention).

Key Definitions

  • Redundancy: Duplication of the same data value across multiple tuples or multiple relations.
  • Insertion anomaly: Inability to insert a tuple because required information is missing, even though the core entity is valid.
  • Deletion anomaly: Unintended loss of information when deleting a tuple because unrelated facts are stored together.
  • Update anomaly: Inconsistency caused by updating some but not all copies of a redundant data item.
  • Functional dependency: A constraint that states that the value of one set of attributes uniquely determines the value of another set of attributes.
  • Lossless join decomposition: A decomposition that allows the original relation to be reconstructed exactly by taking the natural join of the decomposed relations, without generating spurious tuples.
  • Atomic domain: A domain whose elements are considered indivisible units; no structure is visible to the DBMS.
  • First normal form (1NF): A relation in which the domain of every attribute is atomic and every attribute contains only a single value (no repeating groups).

Common Mistakes

  • Misconception: "Redundancy is harmless as long as storage is cheap."
    (Correction: The primary problem with redundancy is not storage cost but anomalies (insertion, deletion, update). Cheap storage does not fix inconsistent data.)
  • Misconception: "A good design is always the one with the fewest tables."
    (Correction: Fewer tables often mean more redundancy. A properly normalised design usually has more tables, but each table represents a single concept, reducing anomalies.)
  • Misconception: "Functional dependencies can be discovered by looking at the current data."
    (Correction: Functional dependencies are business rules that must hold for all possible instances, not just the current snapshot. Observing that two values are equal in the current data does not prove a dependency.)

Edge Cases

  • Redundancy by design: In data warehousing, derived and aggregated data are intentionally stored redundantly to speed up analytical queries. This is a conscious trade‑off, not a design flaw.
  • Lossy decomposition that produces more tuples: Some students think a lossy decomposition always produces fewer tuples. In reality, the natural join of a lossy decomposition may produce extra spurious tuples (more than the original), as shown in the supplier‑parts example.

Real-world Application

Every commercial RDBMS (Oracle, MySQL, PostgreSQL, SQL Server) relies on normalisation theory. When a developer creates a table with repeating groups (e.g., multiple phone numbers in one column), the database does not enforce 1NF, but the application suffers from anomalies. Real‑world e‑commerce platforms, banking systems, and ERP systems all undergo rigorous normalisation during design to maintain data integrity.

Key Takeaways

* A good design avoids redundancy, supports integrity, and is easy to understand.
* Redundancy leads to insertion, deletion and update anomalies – not just wasted space.
* Functional dependencies capture business rules and are essential for normalisation.
* Decompositions must be lossless‑join to preserve information; lossy decompositions create spurious tuples.
* First normal form (1NF) requires atomic domains and single‑valued attributes – the foundation for all higher normal forms.

2. Redundancy and Anomalies

Definition

Redundancy occurs when the same piece of information is stored in multiple places within a database. Anomalies are undesirable side‑effects (inconsistencies) that arise when performing insert, delete or update operations on a relation that contains redundant data.

Explanation

The lecture uses the classic example of merging instructor and department into a single table instructor_with_dept:

ID Name Dept_name Salary Building Budget
10101 Einstein Physics 95000 Watson 500000
12121 Gold Physics 87000 Watson 500000

Here Building and Budget depend only on Dept_name. Because two instructors belong to the same department, those values are repeated – a classic redundancy.

Insertion anomaly: Suppose we want to add a new department ‘Computer Science’ that has no instructors yet. With the combined table, we cannot insert the department record because the primary key ID would be missing. We are forced to add a dummy instructor, which is not allowed (or creates meaningless data).

Deletion anomaly: If we delete the last instructor of a department, we lose all information about that department (its building and budget) even though the department still exists in reality.

Update anomaly: If the budget of the Physics department changes, we must update every tuple that contains Physics. If we miss one, the database becomes inconsistent (different budget values for the same department).

These anomalies are eliminated by decomposing the table into two separate relations: instructor(ID, name, dept_name, salary) and department(dept_name, building, budget). Now each fact is stored exactly once.

Intuition / Mental Model

Imagine a spreadsheet where a department’s budget appears on every row for instructors of that department. Updating the budget means finding every occurrence. If you miss one, two instructors from the same department show different budgets – confusion. Splitting into two sheets (one for instructors, one for departments) and linking them by department name makes updates single‑place and eliminates repetition.

Key Definitions

  • Insertion anomaly: A situation where a valid fact cannot be recorded because other required facts are not yet available.
  • Deletion anomaly: Unintended loss of facts when deleting a tuple that also holds unrelated information.
  • Update anomaly: Inconsistency that occurs when the same data is stored multiple times and not all copies are updated.

Formulas (if any)

Not applicable, but the condition for redundancy can be expressed as: ∃ a functional dependency \(X \rightarrow Y\) where \(X\) is not a superkey and the table contains multiple tuples with the same \(X\) but different unrelated attributes.

Common Mistakes

  • Misconception: "Only the largest tables suffer from anomalies."
    (Correction: Even small tables can have anomalies if they combine two independent entity types. The size is irrelevant; the presence of non‑key functional dependencies matters.)
  • Misconception: "Anomalies can be fixed by adding triggers."
    (Correction: While triggers can enforce consistency, they do not remove the structural redundancy. Normalisation is the proper theoretical solution. Triggers are only a band‑aid.)

Edge Cases

  • Snapshot tables: Historical or audit tables deliberately store redundant data to preserve past states. In such cases anomalies are not considered harmful because the goal is to freeze a point in time.
  • Materialised views: They store redundant aggregated data but are refreshed automatically; anomalies are avoided by controlled refresh policies.

Real-world Application

Every transactional system (e.g., online banking, flight booking) applies normalisation to prevent anomalies. For instance, a flight reservation system stores passenger details separately from flight schedules; otherwise changing a flight time would require updating every passenger’s record.

Key Takeaways

* Redundancy is not just about storage – it is the root cause of insertion, deletion and update anomalies.
* Combined tables that mix independent concepts are highly prone to anomalies.
* Decomposing based on functional dependencies eliminates redundancy.
* Always test a design by asking: “What happens when I insert/delete/update a single real‑world fact?”
* The anomalies are formally defined in the lecture and are standard exam topics.

3. Functional Dependencies – Formal Introduction

Definition

A functional dependency (FD) is a constraint between two sets of attributes in a relation. It is written as \(X \rightarrow Y\) (read “\(X\) functionally determines \(Y\)”) and means that for any two tuples \(t_1\) and \(t_2\) in any legal instance of the relation, if \(t_1[X] = t_2[X]\) then it must be that \(t_1[Y] = t_2[Y]\). In other words, the value of \(X\) uniquely determines the value of \(Y\).

Explanation

Functional dependencies are the formal way to capture business rules. In the instructor‑department example, the business rule “a department has a unique building and a unique budget” is expressed as:

\[ \text{dept\_name} \rightarrow \text{building, budget} \]

This FD holds because if two tuples have the same dept_name, they must agree on building and budget. Conversely, the FD \(\text{ID} \rightarrow \text{dept\_name}\) holds because an instructor belongs to exactly one department.

FDs are a generalisation of the concept of key. A superkey \(K\) is a set of attributes that functionally determines all attributes of the relation: \(K \rightarrow R\).

It is crucial to understand that FDs are semantic constraints derived from the real‑world meaning of data, not from a particular instance. The lecture emphasises that you cannot infer an FD just by looking at a few rows; you need domain knowledge.

Intuition / Mental Model

Think of an FD as a mathematical function: \(X\) is the input, \(Y\) is the output. If you know the input, you can compute the output uniquely. In a database, “input” means the value of attributes in \(X\), and “output” means the value of attributes in \(Y\).

Key Definitions

  • Functional dependency (FD): \(X \rightarrow Y\) – each \(X\) value is associated with exactly one \(Y\) value.
  • Trivial FD: An FD where the right‑hand side is a subset of the left‑hand side (e.g., \(AB \rightarrow A\)). Always true by definition.
  • Non‑trivial FD: An FD where the right‑hand side is not a subset of the left‑hand side.
  • Closure of a set of FDs (F⁺): The set of all FDs that can be derived from the given set using Armstrong’s axioms.
  • Attribute closure (\(X^+\)): The set of all attributes that are functionally determined by \(X\) under a given set of FDs.

Formulas (if any)

\[ \text{For any instance } r \text{ of } R,\ \forall t_1,t_2 \in r,\ (t_1[X] = t_2[X]) \Rightarrow (t_1[Y] = t_2[Y]) \]

Common Mistakes

  • Misconception: "If the current data shows unique values in Y for each X, then \(X \rightarrow Y\) holds."
    (Correction: The current instance might accidentally satisfy the FD, but the FD must hold for all possible future instances. Therefore you cannot “discover” FDs solely from data; you need the business rules.)
  • Misconception: "\(X \rightarrow Y\) implies \(Y \rightarrow X\)."
    (Correction: Functional dependencies are not symmetric. \(X \rightarrow Y\) says nothing about whether \(Y\) determines \(X\).)
  • Misconception: "All FDs are trivial."
    (Correction: Only FDs where RHS ⊆ LHS are trivial. Most real‑world constraints are non‑trivial.)

Edge Cases

  • Empty left‑hand side: \(\emptyset \rightarrow Y\) means that all tuples must have the same value for \(Y\) (i.e., \(Y\) is constant across the relation).
  • FDs with composite left‑hand side: \(AB \rightarrow C\) means the combination of \(A\) and \(B\) determines \(C\). Even if \(A\) alone does not determine \(C\), the combination might.

Real-world Application

FDs are used in database design tools (e.g., Oracle Designer, ERwin) to automatically normalise schemas. They also form the basis for query optimisation: if the system knows that \(A \rightarrow B\), it can avoid duplicate checks.

Key Takeaways

* An FD \(X \rightarrow Y\) states that \(X\) uniquely determines \(Y\).
* FDs are semantic constraints, not statistical observations.
* Every superkey corresponds to an FD \(K \rightarrow R\).
* FDs are the foundation for normalisation – they tell us when to decompose.

4. Decomposition – Lossless vs Lossy Join

Definition

Decomposition is the process of replacing a relation \(R\) with a set of relations \(\{R_1, R_2, \dots, R_n\}\) such that each \(R_i\) is a subset of the attributes of \(R\) (a projection) and every attribute of \(R\) appears in at least one \(R_i\). The decomposition is lossless‑join if, for every legal instance \(r\) of \(R\), the natural join of the projections of \(r\) onto the \(R_i\) yields exactly \(r\) (no spurious tuples).

Explanation

The lecture contrasts two decompositions of the instructor_with_dept table:

  • Good (lossless) decomposition: \(\{ \text{instructor(ID, name, dept\_name, salary)}, \text{department(dept\_name, building, budget)} \}\). The natural join on dept_name reconstructs the original table exactly because dept_name is a key in the department relation.
  • Bad (lossy) decomposition: e.g., decomposing into \(\text{emp1(ID, name)}\) and \(\text{emp2(name, salary)}\) – if two employees share the same name, the join produces spurious combinations.

A sufficient condition for a binary decomposition \(R_1, R_2\) to be lossless is:

\[ R_1 \cap R_2 \rightarrow R_1 \quad \text{or} \quad R_1 \cap R_2 \rightarrow R_2 \]

That is, the common attributes must form a superkey for at least one of the decomposed relations.

The lecture gives the supplier‑parts example where decomposing on quantity (not a key) leads to a lossy join (extra tuples appear).

Intuition / Mental Model

Think of a puzzle: you cut a picture into two pieces. If the cut follows a natural boundary (like a river), you can glue them back without distortion. If you cut randomly, you might create extra pieces that do not fit. The lossless condition ensures the cut follows a “key” boundary so that the pieces fit uniquely.

Key Definitions

  • Lossless‑join decomposition: A decomposition that allows exact reconstruction via natural join.
  • Lossy decomposition: A decomposition that either loses information or creates spurious tuples upon join.
  • Spurious tuples: Tuples that appear in the join but were not present in the original relation.

Formulas (if any)

For binary decomposition \(R_1\) and \(R_2\) of \(R\):

\[ \text{Decomposition is lossless} \iff (R_1 \cap R_2) \rightarrow R_1 \ \text{or}\ (R_1 \cap R_2) \rightarrow R_2 \]

Equivalently, \( (R_1 \cap R_2) \) is a superkey in either \(R_1\) or \(R_2\).

Common Mistakes

  • Misconception: "Any decomposition that does not lose attributes is lossless."
    (Correction: Attribute preservation is necessary but not sufficient. The join must not create extra rows.)
  • Misconception: "Lossless decomposition always produces fewer tuples."
    (Correction: Lossless decomposition produces exactly the original tuples after join. Lossy decomposition often produces more tuples.)
  • Misconception: "If the intersection is empty, the decomposition is lossless."
    (Correction: If the intersection is empty, the join becomes a Cartesian product, which almost always produces spurious tuples – it is lossy unless one of the relations is empty.)

Edge Cases

  • If the original relation contains no tuples (empty instance), any decomposition is technically lossless, but the definition requires it to hold for all instances, so the condition must be satisfied for non‑empty cases as well.
  • When the intersection is a candidate key in both relations, the decomposition is still lossless (the condition is satisfied twice).

Real-world Application

When a DBA splits a large table into two for performance reasons (vertical partitioning), they must ensure the split is lossless, typically by keeping the primary key in both partitions. This is common in column‑oriented databases.

Key Takeaways

* Lossless join is mandatory for any decomposition – otherwise information is lost.
* The condition for binary lossless decomposition is that the common attributes form a superkey in at least one of the resulting relations.
* Always test decomposition by constructing natural joins on sample data (including edge cases).
* Lossless decomposition is necessary but not sufficient for a good design; dependency preservation is also required.

5. Atomic Domains and First Normal Form (1NF)

Definition

A domain is atomic if elements of the domain are considered indivisible units. A relation schema \(R\) is in First Normal Form (1NF) if the domains of all its attributes are atomic and every attribute contains only a single value (no repeating groups or multi‑valued attributes).

Explanation

In the relational model, every attribute value must be a single, indivisible value. Composite attributes (e.g., Name composed of FirstName and LastName) violate atomicity. Multi‑valued attributes (e.g., a set of phone numbers) also violate 1NF because a single tuple would contain multiple values.

The lecture gives the example of a Customer relation:

Customer( custID, name, phone_numbers )

If a customer has two phone numbers, the naive representation either stores both numbers in one attribute (comma‑separated string, not atomic) or creates two columns phone1, phone2 (still violates 1NF because the number of columns is fixed while the number of phones can vary).

The correct way to achieve 1NF is to decompose into two relations:

\[ \text{Customer(custID, name)} \]

\[ \text{CustomerPhone(custID, phone)} \]

Now each phone number is stored in a separate tuple, and every attribute is atomic.

Intuition / Mental Model

1NF is like a spreadsheet where every cell contains a single number or a simple text, not a list. If you ever feel tempted to put a comma‑separated list inside a cell, you are violating 1NF.

Key Definitions

  • Atomic domain: A domain where the DBMS cannot see any internal structure; the value is treated as a whole.
  • Repeating group: A set of values that can appear multiple times for a single tuple, e.g., multiple phone numbers.
  • First normal form (1NF): A relation that contains no repeating groups and all attributes are atomic.

Formulas (if any)

Not applicable, but note that 1NF is the starting point for all higher normal forms (2NF, 3NF, BCNF, etc.).

Common Mistakes

  • Misconception: "1NF prohibits composite attributes completely."
    (Correction: The requirement is that the DBMS should treat the attribute as atomic. If you split Name into FirstName and LastName as separate attributes, that is fine – each is atomic.)
  • Misconception: "Using a fixed number of columns (phone1, phone2, phone3) is 1NF."
    (Correction: While each column contains a single value, the design is not 1NF because the column set is repeating. The correct solution is a separate table.)
  • Misconception: "1NF is always desirable."
    (Correction: 1NF is mandatory in the classical relational model. However, modern NoSQL databases intentionally relax 1NF for flexibility.)

Edge Cases

  • Codes with embedded meaning: e.g., CourseID = "CS101" where "CS" means department. The domain is still atomic if the DBMS does not interpret substrings; decomposition is a design choice, not a 1NF violation.
  • JSON/XML data types: Some DBMSs provide native JSON types; storing a JSON array in a column violates 1NF from the relational perspective, but the system may treat it as an atomic document. This is an exception in modern hybrid systems.

Real-world Application

Every SQL database that claims to be relational (MySQL, PostgreSQL, Oracle, SQL Server) enforces 1NF at the schema level – you cannot have a column that contains multiple values unless you use a complex data type (which is often discouraged).

Key Takeaways

* 1NF requires atomic domains and single‑valued attributes.
* Repeating groups (multiple phone numbers) must be moved to a separate table.
* Violations of 1NF lead to redundancy and anomalies.
* The standard approach to bring a relation to 1NF is decomposition.
* All subsequent normalisation (2NF, 3NF, BCNF) assume 1NF as a starting point.

6. Converting Non‑1NF Relations to 1NF

Definition

Converting a non‑1NF relation to 1NF involves eliminating repeating groups and making every attribute atomic. The standard method is to create a separate relation for the multi‑valued attribute, with the original key (or a foreign key) as part of the new relation’s key.

Explanation

Consider the Customer relation with a multi‑valued attribute phones:

Customer( custID, name, phones )

Step‑by‑step conversion:

  1. Create a new relation CustomerPhone that contains the primary key of Customer (as a foreign key) plus the attribute that was multi‑valued.
  2. For each phone number of a customer, insert one tuple into CustomerPhone.
  3. Remove the phones attribute from Customer.

Result:

Customer( custID, name )
CustomerPhone( custID, phone )

Both relations are in 1NF. The key of CustomerPhone is (custID, phone).

The lecture also shows the alternative of using multiple columns (phone1, phone2) and explains why that is unacceptable because the number of phones is not fixed and queries become awkward.

Intuition / Mental Model

Imagine a library card where each member can have multiple phone numbers. Instead of writing all numbers on one line (separated by commas), you give each member a separate sheet of paper that lists one number per line. The member’s ID connects the two sheets.

Key Definitions

  • Repeating group: A set of values that can appear zero or more times for a single tuple.
  • Flattening: The process of converting a multi‑valued attribute into a separate table.

Common Mistakes

  • Misconception: "Using a string with delimiters (e.g., '555-1234,555-5678') is acceptable because it's a single value."
    (Correction: Although it is a single string, the domain is not atomic because queries that need to extract individual numbers become difficult and inefficient. This violates the spirit of 1NF.)
  • Misconception: "Creating a separate table for phone numbers is over‑normalisation."
    (Correction: It is the standard relational solution. Without it, updating a phone number would require parsing a string, leading to anomalies.)

Edge Cases

  • Multiple multi‑valued attributes: Each multi‑valued attribute gets its own separate table.
  • Nested repeating groups: A relation that has a repeating group within a repeating group – handled by repeated decomposition.

Real-world Application

Every time you design a database for a customer relationship management (CRM) system, you create separate tables for addresses, phone numbers, email addresses – all to keep the customer table in 1NF.

Key Takeaways

* Non‑1NF relations are normalised by decomposing the multi‑valued part into a separate table.
* The key of the new table includes the original key plus the multi‑valued attribute.
* Using multiple columns (phone1, phone2) is not a valid 1NF solution.
* 1NF conversion is the first step before applying higher normal forms.
* Always ensure every column contains atomic, single values.
