1. Good Design Goals

Definition

Good database design is a set of principles and practices that ensure the resulting relational schema accurately represents the real‑world domain, avoids redundancy, supports data integrity, provides efficient access, and is easy to understand and maintain.

Explanation

A database is not just a storage box; it is a model of some part of the real world (e.g., a university, a bank, a hospital). The design must reflect the actual structure and constraints of that world. For example, if in reality each department has exactly one building, the design must enforce that. If the real world changes, the design should be extensible.

Because a database will be used over a long period, the design must handle all expected data that may appear in the future. This is called adequacy over time. A design that works for 100 records might break when 1 million records are added if indexing or normalization is poor.

Redundant storage (storing the same fact multiple times) wastes disk space, but more importantly, it leads to anomalies when data is updated. A good design aims to eliminate redundancy as much as possible, without harming other goals.

Efficiency is also crucial: queries should return results quickly. However, there is often a trade‑off: removing all redundancy (normalizing) sometimes increases the number of joins, which can slow down reads. Therefore, designers sometimes intentionally introduce some redundancy for performance reasons (denormalization).

Integrity constraints (like foreign keys, check constraints, and functional dependencies) must be supportable. A good design makes it easy to enforce these constraints within the database itself, not in application code.

Finally, the design must be clean, consistent, and easy to understand. Other developers (and your future self) will need to maintain it. Clear naming, consistent use of keys, and following normal forms help achieve this.

Intuition / Mental Model

Think of designing a database like designing a house. You want it to match the blueprints (real world), be able to accommodate future furniture (adequacy), avoid duplicate wiring or pipes (redundancy), be easy to move around (efficient access), have strong structural integrity (integrity), and be simple enough that any electrician can understand it (clean design). Sometimes you might add an extra socket (redundancy) to avoid having to run a long extension cord (joins) – that's a trade‑off.

Key Definitions

  • Adequacy: The ability of a design to represent all expected data over time.
  • Redundancy: Storing the same data multiple times in a database.
  • Anomaly: An inconsistency or problem that can arise when modifying redundant data.
  • Denormalization: Intentionally adding redundancy to improve read performance.
  • Integrity Constraint: A rule that data must satisfy (e.g., NOT NULL, UNIQUE, FOREIGN KEY).

Formulas (if any)

There is no formula, but the trade‑off can be expressed as:
\[ \text{Performance} = \frac{\text{Read speed}}{\text{Write overhead}} \quad \text{but with normalization, writes become faster, reads may become slower.} \]

Common Mistakes

  • Misconception 1: "Normalization always improves performance."
    (Correction: Normalization reduces redundancy, which improves update performance and eliminates anomalies, but it can slow down complex read queries that require many joins.)
  • Misconception 2: "Redundancy is only a storage issue."
    (Correction: The bigger problem is anomalies. Storage is cheap; data inconsistency is expensive.)
  • Misconception 3: "A good design must satisfy all goals equally."
    (Correction: Goals often conflict. Designers must make trade‑offs based on application needs.)

Edge Cases

  • Read‑only data warehouses: high redundancy can be acceptable because no updates occur, so anomalies are not a risk.
  • Very high‑frequency transaction systems: may sacrifice some normalization to avoid excessive join overhead.

Real‑world Application

E‑commerce platforms like Amazon use highly normalized schemas for order processing (to avoid anomalies) but may use denormalized caches for product recommendations (to improve speed).

Key Takeaways

• Good design goals: real‑world reflection, adequacy, no redundancy, efficient access, integrity support, clean & understandable.
• Redundancy leads to anomalies, not just wasted space.
• There is a trade‑off between normalization (eliminates anomalies) and query performance (may require joins).
• Denormalization is sometimes used deliberately.
• Always design with future data and maintenance in mind.

2. Redundancy and Anomalies

Definition

Redundancy occurs when the same data item is stored multiple times in a database. Anomalies are problems that arise when inserting, deleting, or updating redundant data, leading to inconsistent or incomplete information.

Explanation

Consider a single table that combines instructor and department information: instructor(ID, name, dept_name, salary, building, budget). If a department has multiple instructors, the building and budget are repeated for each instructor. That is redundancy.

Insertion anomaly: You cannot add a new department that has no instructors yet because the table requires an instructor ID (primary key). You would have to insert a dummy instructor or leave fields null, which violates design principles.

Deletion anomaly: If you delete the last instructor of a department, you also lose the department's building and budget information, even though the department still exists in reality.

Update anomaly: If the budget of a department changes, you must update every row containing that department. If you miss one, the database becomes inconsistent – some rows show the old budget, some the new.

These anomalies are the primary motivation for normalization.

Intuition / Mental Model

Imagine a spreadsheet where you list each sale together with the customer's full address. If a customer moves, you have to update every row where that customer appears. That's an update anomaly. If you delete the last sale of a customer, you lose their address – deletion anomaly. You cannot add a new customer without a sale – insertion anomaly. The solution is to separate customers and sales into two tables.

Key Definitions

  • Insertion anomaly: Inability to insert a record without adding unrelated data.
  • Deletion anomaly: Unintended loss of information when deleting a record.
  • Update anomaly: Inconsistency caused by updating some but not all copies of redundant data.

Formulas / Examples

Example table (instructor + department):

ID name dept_name salary building budget
101 Einstein Physics 95000 Watson 500000
102 Gold Physics 87000 Watson 500000

Redundancy: building and budget repeated. Update anomaly: changing budget requires updating both rows. Insertion anomaly: cannot add a new department "Math" without an instructor. Deletion anomaly: deleting both instructors loses Physics department info.

Common Mistakes

  • Misconception: "Anomalies only happen with very large data." (Correction: Anomalies can happen even with two rows, as shown above.)
  • Misconception: "Null values solve insertion anomalies." (Correction: Nulls can be used but they complicate queries and are not a clean solution; they also violate entity integrity if the primary key is null.)

Edge Cases

  • If a department has zero instructors, the combined table cannot represent it at all – insertion anomaly is absolute.
  • If a department has only one instructor, deletion anomaly still occurs because deleting that one row loses the department.

Real‑world Application

Banking systems avoid anomalies by separating customer accounts from transaction histories. A customer's address is stored once, not repeated in every transaction row.

Key Takeaways

• Three anomalies: insertion, deletion, update.
• Redundancy is the root cause.
• Anomalies lead to data inconsistency and loss of information.
• Normalization eliminates anomalies by decomposing tables.
• Always test your design with example data to detect potential anomalies.

3. Decomposition and Lossless Join

Definition

Decomposition is the process of splitting a relation into two or more smaller relations. A decomposition is lossless join if, after decomposing, the natural join of the smaller relations yields exactly the original relation (no spurious tuples and no lost tuples).

Explanation

When we decompose a table to remove redundancy, we must ensure we can later reconstruct the original information. If we decompose carelessly, we may get extra tuples when joining back – this is called a lossy decomposition (or lossy join).

Example: Decompose the instructor‑department table into instructor1(ID, name, dept_name, salary) and department(dept_name, building, budget). This decomposition is lossless because the common attribute dept_name is a key in the department relation. When we join, we recover exactly the original rows.

However, if we decompose into R1(ID, name, building) and R2(dept_name, salary, budget) with no common attribute, the join would be a Cartesian product, producing many spurious rows – lossy.

The formal condition for a binary decomposition (R into R1 and R2) to be lossless is that the intersection of R1 and R2 must be a superkey of either R1 or R2 (or both).

Intuition / Mental Model

Imagine a jigsaw puzzle. Decomposing is like breaking the puzzle into pieces. Lossless decomposition means you can reassemble the puzzle perfectly. Lossy decomposition means you end up with extra pieces that don't belong or missing pieces.

Key Definitions

  • Lossless join decomposition: Join of decomposed relations returns the original relation.
  • Lossy decomposition: Join returns more tuples than the original (spurious tuples).
  • Superkey: A set of attributes that uniquely identifies a tuple.

Formulas

For R decomposed into R1 and R2:
Let \( I = R1 \cap R2 \).
The decomposition is lossless if \( I \rightarrow R1 \) or \( I \rightarrow R2 \) (i.e., I is a superkey of R1 or R2).

Common Mistakes

  • Misconception: "Any decomposition that removes redundancy is lossless." (Correction: Not true; you must check the join condition.)
  • Misconception: "If the intersection is a key in both, it's lossless." (Correction: It's sufficient that it is a key in at least one.)
  • Misconception: "Lossless join is the only property needed." (Correction: Dependency preservation is also important, but lossless join is mandatory.)

Edge Cases

  • If the intersection is empty, the decomposition is always lossy (unless one of the relations is empty, which is trivial).
  • If the intersection is a superkey of both, it's still lossless.

Real‑world Application

Online shopping carts: the cart table is often decomposed into cart_items and products. Joining them must produce the exact cart content – lossless decomposition is critical.

Key Takeaways

• Decomposition must be lossless to preserve information.
• For binary decomposition, the common attribute(s) must be a superkey of at least one side.
• Lossy decompositions create spurious tuples and are unacceptable.
• Always verify lossless property before implementing a decomposition.
• Lossless join is necessary but not sufficient for a good design (dependency preservation also matters).

4. First Normal Form (1NF)

Definition

A relation is in First Normal Form (1NF) if the domain of every attribute is atomic (indivisible) and every attribute contains only a single value (no multi‑valued attributes).

Explanation

Atomic means that the attribute cannot be broken down into smaller meaningful parts within the database. For example, a person's name can be split into first name, middle name, last name – so storing a single "name" attribute violates 1NF if you need to access parts separately. However, if you treat the whole name as an indivisible string, it can be atomic – atomicity depends on usage.

Multi‑valued attributes (e.g., a person having multiple phone numbers) also violate 1NF. The standard solution is to create a separate table for the multi‑valued attribute, linked by a foreign key.

All relational database management systems (RDBMS) require tables to be in 1NF by design – you cannot create a table with a column that contains a set of values directly. However, you could violate 1NF by using delimited strings (e.g., "111-222,333-444") but that is strongly discouraged.

Intuition / Mental Model

Think of each cell in a table as a single, indivisible piece of data – like a number or a simple text string. If a cell can contain a list or a composite value (like "John Smith" where you sometimes need "John" separately), then you are not in 1NF.

Key Definitions

  • Atomic domain: A domain whose elements are considered indivisible by the database.
  • Multi‑valued attribute: An attribute that can have multiple values for a single entity (e.g., phone numbers).
  • Composite attribute: An attribute composed of multiple smaller attributes (e.g., address as street, city, zip).

Formulas / Examples

Not in 1NF:

CustomerID Name PhoneNumbers
1 John Doe 555-1234, 555-5678

Converted to 1NF (separate table):

Customer(1, John Doe) and CustomerPhone(1, 555-1234), (1, 555-5678).

Common Mistakes

  • Misconception: "Any string is atomic." (Correction: Atomicity depends on usage. If you extract substrings in queries, it's not atomic.)
  • Misconception: "Using multiple columns (Phone1, Phone2) solves the multi‑value problem." (Correction: It limits the number of values, wastes space with nulls, and makes queries complex – separate table is better.)

Edge Cases

  • Course codes like "CS101" – if you never need to extract "CS", it can be considered atomic; otherwise, split into department_code and course_number.
  • JSON/XML columns in modern databases – they are not atomic in the relational sense, but some systems treat them as special types. For normalization, they violate 1NF.

Real‑world Application

Every properly designed relational database automatically ensures 1NF by using separate tables for multi‑valued relationships (e.g., a "Tags" table linked to a "Posts" table in a blog).

Key Takeaways

• 1NF requires atomic, single‑valued attributes.
• All tables in a standard RDBMS are in 1NF by definition.
• Multi‑valued attributes must be moved to a separate table.
• Composite attributes should be split if their parts are queried individually.
• 1NF is the foundation for all higher normal forms.
