14. Extraneous Attributes

Definition

An attribute in a functional dependency is extraneous if it can be removed from the left‑hand side or the right‑hand side without changing the closure \( F^+ \) of the set of FDs. Removing an extraneous attribute yields a simpler, equivalent set of FDs.

Explanation

There are two cases:

  • Left‑side extraneous: Given \( \alpha \rightarrow \beta \), an attribute \( A \in \alpha \) is extraneous if \( (\alpha - \{A\}) \rightarrow \beta \) can be inferred from \( F \). In other words, the FD still holds without \( A \) on the left.
  • Right‑side extraneous: Given \( \alpha \rightarrow \beta \), an attribute \( B \in \beta \) is extraneous if \( \alpha \rightarrow (\beta - \{B\}) \) can be inferred from \( F \).

To test left‑side extraneous, compute \( (\alpha - \{A\})^+ \) using \( F \). If it contains \( \beta \), then \( A \) is extraneous.

To test right‑side extraneous, compute \( \alpha^+ \) using \( F' = (F - \{\alpha \rightarrow \beta\}) \cup \{\alpha \rightarrow (\beta - \{B\})\} \). If it contains \( B \), then \( B \) is extraneous.

Extraneous attributes are removed during the computation of a canonical cover.

Intuition / Mental Model

Think of an FD as a “rule”. If you can prove the rule with a smaller left side, the extra attribute on the left is just “baggage”. Similarly, if the rule still holds when you drop an attribute from the right side, that attribute was not really determined – it might be redundant because of other FDs.

Key Definitions

  • Extraneous attribute: An attribute that can be removed without changing the closure.
  • Left‑side extraneous: The FD is still valid without that attribute on LHS.
  • Right‑side extraneous: The FD is still valid without that attribute on RHS.

Examples

Example 1 (left‑side extraneous): \( F = \{ AB \rightarrow C, A \rightarrow C \} \). In \( AB \rightarrow C \), is \( B \) extraneous? Compute \( A^+ \) using F: \( A^+ = \{A, C\} \) (from \( A \rightarrow C \)). Does it contain \( C \)? Yes. So \( A \rightarrow C \) already holds, so \( B \) is extraneous. We can remove \( B \) and keep only \( A \rightarrow C \).

Example 2 (right‑side extraneous): \( F = \{ A \rightarrow BC, A \rightarrow B \} \). In \( A \rightarrow BC \), is \( C \) extraneous? Check if \( A \rightarrow B \) alone implies \( A \rightarrow BC \)? No. But we need to test using the definition: Remove \( C \) from RHS, get \( A \rightarrow B \). Does \( F \) logically imply \( A \rightarrow BC \)? Actually we need to see if \( A \rightarrow C \) is implied. Compute \( A^+ \) with \( F \) (using both FDs): \( A^+ = \{A, B, C\} \). Since \( C \) is in \( A^+ \), \( A \rightarrow C \) holds. Therefore, \( A \rightarrow BC \) can be replaced by \( A \rightarrow B \) and \( A \rightarrow C \) (decomposition), but \( A \rightarrow C \) is already implied? Wait, careful: Right‑side extraneous means that after removing the attribute, the new FD set (with the reduced FD) is equivalent to the original. Here, if we replace \( A \rightarrow BC \) with \( A \rightarrow B \), we lose \( A \rightarrow C \). But if \( A \rightarrow C \) is implied by other FDs (it is, from \( A \rightarrow B \) and ... no, we don't have \( B \rightarrow C \)). So in this example, \( C \) is not extraneous. A correct example: \( F = \{ A \rightarrow B, B \rightarrow C, A \rightarrow BC \} \). In \( A \rightarrow BC \), \( C \) is extraneous because \( A \rightarrow B \) and \( B \rightarrow C \) imply \( A \rightarrow C \). So the set without \( A \rightarrow BC \) is equivalent.

Common Mistakes

  • Misconception: “Extraneous means the attribute is never used.” (Correction: It may be used but can be removed without loss.)
  • Misconception: “Removing extraneous attributes always yields a smaller FD set.” (Correction: It reduces the size of individual FDs; the number of FDs may remain same or decrease after union.)

Key Takeaways

• Extraneous attributes are removed to obtain a minimal (canonical) cover.
• Left‑side extraneous: test using \( (\alpha - A)^+ \).
• Right‑side extraneous: test using modified set.
• Only non‑trivial FDs should be considered.

15. Canonical Cover (Minimal Set)

Definition

A canonical cover \( F_c \) for a set of FDs \( F \) is a minimal set of FDs that is equivalent to \( F \) (i.e., \( F_c^+ = F^+ \)) and satisfies:

  1. No FD in \( F_c \) contains an extraneous attribute.
  2. The left‑hand sides of all FDs in \( F_c \) are unique (no two FDs have the same left side).

Explanation

Computing a canonical cover is important for normalization algorithms (e.g., 3NF synthesis) and for eliminating redundancy in the specification of constraints. The algorithm:

  1. Use union rule to combine FDs with the same left side (e.g., \( \alpha \rightarrow \beta \) and \( \alpha \rightarrow \gamma \) become \( \alpha \rightarrow \beta\gamma \)).
  2. For each FD \( \alpha \rightarrow \beta \), check each attribute in \( \alpha \) (left) for extraneousness. Remove if extraneous.
  3. For each FD \( \alpha \rightarrow \beta \), check each attribute in \( \beta \) (right) for extraneousness. Remove if extraneous.
  4. If any change was made, repeat from step 1 (because removing attributes may create new opportunities for union or further reduction).

The result is a canonical cover. Note that canonical covers are not unique; the order of processing can lead to different but equivalent minimal sets.

Intuition / Mental Model

Think of the canonical cover as the “tightest” set of rules that capture all constraints. No rule has unnecessary parts, and no two rules say the same thing with different left sides – they are combined.

Key Definitions

  • Canonical cover: Minimal, equivalent set of FDs with unique left sides and no extraneous attributes.
  • Minimal set: Same as canonical cover.

Example

\( F = \{ A \rightarrow B, B \rightarrow C, A \rightarrow C, AB \rightarrow C \} \).

  1. Combine same left side: \( A \rightarrow BC \) (from \( A \rightarrow B \) and \( A \rightarrow C \)). So F becomes \( \{ A \rightarrow BC, B \rightarrow C, AB \rightarrow C \} \).
  2. Check \( AB \rightarrow C \): Is \( A \) extraneous? Compute \( B^+ \) using F: \( B^+ = \{B, C\} \). Does it contain \( C \)? Yes. So \( B \rightarrow C \) already holds, thus \( A \) is extraneous on left. Remove A: \( B \rightarrow C \) already exists, so \( AB \rightarrow C \) is redundant and can be removed entirely.
  3. Now F = \( \{ A \rightarrow BC, B \rightarrow C \} \). Check \( A \rightarrow BC \): Is \( C \) extraneous? Compute \( A^+ \) using only \( A \rightarrow B \) (if we remove C from RHS) and \( B \rightarrow C \)? Actually we need to check if \( A \rightarrow C \) is implied without \( A \rightarrow C \) itself. With \( A \rightarrow B \) and \( B \rightarrow C \), \( A \rightarrow C \) holds. So \( C \) is extraneous on right. Remove C: \( A \rightarrow B \).
  4. Now F = \( \{ A \rightarrow B, B \rightarrow C \} \). Left sides are unique, no extraneous attributes. This is the canonical cover.

Common Mistakes

  • Misconception: “Canonical cover is unique.” (Correction: Different orders of elimination can produce different but equivalent covers.)
  • Misconception: “You must process extraneous attributes on left before right.” (Correction: The algorithm repeats until no change, so order may affect intermediate steps but final result is still a canonical cover.)

Key Takeaways

• Canonical cover = minimal equivalent FD set.
• Steps: union, remove extraneous left, remove extraneous right, repeat.
• Used in 3NF synthesis algorithm.
• Not unique, but all are equivalent.

16. Equivalence of FD Sets

Definition

Two sets of functional dependencies \( F \) and \( G \) are equivalent if \( F^+ = G^+ \), i.e., they imply the same set of FDs. Equivalently, \( F \) covers \( G \) and \( G \) covers \( F \).

Explanation

To test if \( F \) covers \( G \), for each FD \( \alpha \rightarrow \beta \) in \( G \), compute \( \alpha^+ \) using \( F \) and check if \( \beta \subseteq \alpha^+ \). If all FDs in \( G \) are implied by \( F \), then \( F \) covers \( G \). Do the reverse to check equivalence.

Equivalence is important when we transform a set of FDs into a canonical cover – we need to ensure the new set is equivalent to the original.

Intuition / Mental Model

If two sets of rules always give the same conclusions, they are equivalent. Like two different ways to write the same system of equations.

Key Definitions

  • Cover: \( F \) covers \( G \) if every FD in \( G \) is implied by \( F \).
  • Equivalent: \( F \) covers \( G \) and \( G \) covers \( F \).

Example

\( F = \{ A \rightarrow B, B \rightarrow C \} \), \( G = \{ A \rightarrow C, A \rightarrow B, B \rightarrow C \} \). Clearly \( F \) covers \( G \) because \( A \rightarrow C \) is implied by transitivity. And \( G \) covers \( F \) trivially. So they are equivalent.

Common Mistakes

  • Misconception: “If the sets have the same number of FDs, they are equivalent.” (Correction: No, the content matters, not count.)

Key Takeaways

• Equivalence is tested via attribute closure.
• \( F \) covers \( G \) ⇔ for each \( \alpha \rightarrow \beta \) in G, \( \beta \subseteq \alpha^+_F \).
• Canonical cover is equivalent to original set.

Tutorial 5.4: Super Keys and Candidate Keys

Algorithm / Procedure

To find all candidate keys of a relation \( R \) given a set of FDs \( F \):

  1. Start with single attributes. Compute their closure. If closure = all attributes, it is a candidate key.
  2. If no single attribute is a candidate key, try pairs, then triples, etc., but only consider minimal sets that are not already supersets of a candidate key.
  3. Alternatively, use the following systematic method:
    • Find all attributes that never appear on the right side of any FD. These must be in every candidate key (they are “prime” in the sense of mandatory).
    • Start with that mandatory set. Compute its closure. If it is already a superkey, it is the only candidate key (or you need to check minimality).
    • Otherwise, add attributes one by one from the remaining attributes, compute closure, and check for minimal superkey.

To test if a set \( K \) is a candidate key: compute \( K^+ \); if it is all attributes, it is a superkey. Then check if any proper subset of \( K \) is also a superkey. If not, \( K \) is a candidate key.

Worked Example 1

Relation \( R(A,B,C,D) \), FDs: \( A \rightarrow B, B \rightarrow C, C \rightarrow D \).

Compute closures:

  • \( A^+ = \{A, B, C, D\} \) → superkey. Check subsets: no single attribute other than A works. So candidate key = \( \{A\} \).
  • \( B^+ = \{B, C, D\} \) (not all).
  • \( C^+ = \{C, D\} \).
  • \( D^+ = \{D\} \).
  • Pairs like \( B,D \): \( \{B,D\}^+ = \{B,C,D\} \) (no A).

Thus only candidate key is \( A \).

Worked Example 2

Relation \( R(A,B,C) \), FDs: \( AB \rightarrow C, C \rightarrow B \).

  • Compute \( \{A\}^+ = \{A\} \) → no.
  • \( \{B\}^+ = \{B\} \) → no.
  • \( \{C\}^+ = \{C, B\} \) → no.
  • \( \{A,B\}^+ = \{A,B,C\} \) → superkey. Check subsets: \( A^+ \) not superkey, \( B^+ \) not superkey. So \( \{A,B\} \) is candidate key.
  • \( \{A,C\}^+ = \{A,C,B\} \) → superkey. Subset \( A^+ \) not, \( C^+ \) not. So \( \{A,C\} \) is also candidate key.
  • \( \{B,C\}^+ = \{B,C\} \) → no.

Candidate keys: \( \{A,B\} \) and \( \{A,C\} \).

Common Pitfalls

  • Forgetting that candidate keys must be minimal: if a set is a superkey, check its subsets.
  • Assuming that the attribute(s) that appear only on left side must be in every candidate key – true, but be careful: they might not be sufficient alone.

How to verify

After finding candidate keys, ensure that every attribute in the relation is functionally determined by each candidate key. Also ensure that no proper subset of a candidate key is a superkey.

Practice Hints

Use attribute closure repeatedly. Start with single attributes, then move to larger sets only if necessary. For large relations, algorithms exist but manual computation for small schemas is typical in exams.
