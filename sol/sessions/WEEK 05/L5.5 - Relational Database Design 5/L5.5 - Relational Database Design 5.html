Tutorial 5.1: Lossless Join Decomposition

Definition

A decomposition of a relation \( R \) into \( R_1 \) and \( R_2 \) is lossless join if, for every instance \( r \) of \( R \), the natural join of the projections \( \pi_{R_1}(r) \) and \( \pi_{R_2}(r) \) yields exactly \( r \) (no spurious tuples and no missing tuples).

Explanation

Lossless join is mandatory for any decomposition that is to be used in practice; otherwise, information is lost. The sufficient condition for a binary decomposition is:

  • The union of the attributes of \( R_1 \) and \( R_2 \) equals the attributes of \( R \) (i.e., all attributes are covered).
  • The intersection of \( R_1 \) and \( R_2 \) is non‑empty.
  • The intersection attributes form a superkey for either \( R_1 \) or \( R_2 \) (or both).

Formally, if \( I = R_1 \cap R_2 \), then the decomposition is lossless if \( I \rightarrow R_1 \) or \( I \rightarrow R_2 \) (i.e., \( I \) is a superkey of \( R_1 \) or \( R_2 \)).

For decompositions into more than two relations, we can use the chase algorithm or repeatedly apply the binary test.

Intuition / Mental Model

When you split a table and later join it, you must not create rows that did not exist originally. The common attributes act as a “glue”; if that glue uniquely identifies rows on one side, no spurious combinations appear.

Key Definitions

  • Lossless join: Natural join recovers original relation.
  • Lossy decomposition: Join produces extra (spurious) tuples.
  • Superkey condition: Intersection must be a superkey in at least one of the decomposed relations.

Worked Example 1 (Lossless)

\( R(A,B,C) \) with FDs \( A \rightarrow B \). Decompose into \( R_1(A,B) \) and \( R_2(A,C) \).

Intersection = \( \{A\} \). Is \( A \) a superkey of \( R_1 \)? Yes, because \( A \rightarrow B \) holds, so \( A \) determines all attributes of \( R_1 \). Therefore lossless.

Worked Example 2 (Lossy)

\( R(A,B,C) \) with no FDs (or only trivial). Decompose into \( R_1(A,B) \) and \( R_2(B,C) \). Intersection = \( \{B\} \). Is \( B \) a superkey of either? No, because there is no FD that says \( B \) determines \( A \) or \( C \). Hence lossy. Indeed, joining the projections may produce spurious tuples.

Instance: \( R = \{(1,2,3), (4,2,5)\} \). Projections: \( R_1 = \{(1,2), (4,2)\} \), \( R_2 = \{(2,3), (2,5)\} \). Natural join on \( B \) yields \( (1,2,3), (1,2,5), (4,2,3), (4,2,5) \) – two extra tuples.

Common Pitfalls

  • Assuming that if the intersection is a key in the original relation, it is lossless – not sufficient; must be a key in the decomposed relation.
  • Forgetting that the condition is sufficient but not necessary for binary decompositions? Actually it is necessary and sufficient for binary decompositions when FDs are given? The condition (intersection is a superkey of one side) is sufficient; it is also necessary if we require losslessness for all instances, but there are edge cases where it might hold without that condition? In standard theory, the condition is necessary and sufficient for binary decompositions under the assumption that the FDs are the only constraints. We'll stick to the sufficient condition as taught.

How to Verify

Compute the intersection, then compute its closure using the given FDs (applied to the whole schema). If the closure includes all attributes of one of the decomposed relations, then lossless.

Practice Hints

Always list the intersection attributes. Then check if they determine all attributes of \( R_1 \) or all of \( R_2 \). If the intersection is empty, decomposition is always lossy.

18. Dependency Preservation

Definition

A decomposition of \( R \) into \( R_1, R_2, \dots, R_n \) is dependency preserving if the union of the projections of \( F^+ \) onto each \( R_i \) is equivalent to \( F^+ \). In simpler terms, every functional dependency in \( F \) can be checked on the individual decomposed relations without having to perform a join.

Explanation

When we decompose, we lose the ability to directly enforce FDs that involve attributes from different relations unless we take a join (which is expensive). A dependency‑preserving decomposition ensures that each FD is implied by the FDs that hold on the individual relations. This is desirable for performance (constraints can be checked locally).

Testing dependency preservation: For each FD \( \alpha \rightarrow \beta \) in \( F \), compute \( \alpha^+ \) using the FDs that are projected onto the decomposed relations, but using a specialized algorithm that iterates over the relations. Alternatively, use the following approach:

  1. Let \( result = \alpha \).
  2. While \( result \) changes:
    • For each decomposed relation \( R_i \), compute \( result' = (result \cap R_i)^+ \) with respect to \( F \) (or the projected FDs), then take the intersection with \( R_i \) again, and add to \( result \).
  3. If \( \beta \subseteq result \) at the end, then the FD is preserved.

If all FDs in \( F \) are preserved, the decomposition is dependency preserving.

Intuition / Mental Model

Imagine you have a set of rules (FDs). After splitting the data into separate tables, you want each rule to be checkable using only the columns in one table. If a rule requires columns from multiple tables, you have to join them to check it, which is inefficient.

Key Definitions

  • Projection of FDs: For a relation \( R_i \), \( \pi_{R_i}(F) \) = all FDs in \( F^+ \) that involve only attributes from \( R_i \).
  • Dependency preserving: \( F^+ = (\pi_{R_1}(F) \cup \pi_{R_2}(F) \cup \dots \cup \pi_{R_n}(F))^+ \).

Example

\( R(A,B,C) \), \( F = \{ A \rightarrow B, B \rightarrow C \} \). Decompose into \( R_1(A,B) \) and \( R_2(B,C) \).

  • \( \pi_{R_1}(F) = \{ A \rightarrow B \} \).
  • \( \pi_{R_2}(F) = \{ B \rightarrow C \} \).
  • \( A \rightarrow C \) is implied by transitivity but is not in the projection; however, we can check \( A \rightarrow C \) by using \( A \rightarrow B \) on \( R_1 \) and \( B \rightarrow C \) on \( R_2 \). The algorithm would show that it is preserved. So the decomposition is dependency preserving.

Counterexample: \( R(A,B,C) \), \( F = \{ AB \rightarrow C, C \rightarrow A \} \). Decompose into \( R_1(A,C) \) and \( R_2(B,C) \). Intersection = \( C \). Is \( C \) a superkey of \( R_1 \)? \( C \rightarrow A \) holds, so yes, lossless. But is \( AB \rightarrow C \) preserved? The projection on \( R_1 \) gives \( C \rightarrow A \) (and trivial). Projection on \( R_2 \) gives trivial only. There is no way to check \( AB \rightarrow C \) without joining. Hence not dependency preserving.

Common Mistakes

  • Misconception: “If each FD’s attributes are contained in some relation, the decomposition is dependency preserving.” (Correction: Not sufficient; the FD must be implied by the projected FDs, which may require transitivity across relations.)
  • Misconception: “BCNF always preserves dependencies.” (Correction: BCNF may lose dependencies; 3NF is designed to preserve them.)

Key Takeaways

• Dependency preservation avoids expensive joins for constraint checking.
• Test using attribute closure with iterative refinement across relations.
• 3NF decomposition can always be dependency preserving; BCNF may not.
• Lossless join is mandatory; dependency preservation is highly desirable but sometimes sacrificed.

Tutorial 5.5: Dependency Preservation Test

Algorithm / Procedure

Given a decomposition \( R_1, R_2, \dots, R_k \) of \( R \) and a set of FDs \( F \), to test if an FD \( \alpha \rightarrow \beta \) is preserved:

  1. Set \( result = \alpha \).
  2. Repeat until \( result \) does not change:
    1. For each \( R_i \) in the decomposition:
      • Let \( T = result \cap R_i \).
      • Compute \( T^+ \) with respect to \( F \) (the original FD set).
      • Set \( result = result \cup (T^+ \cap R_i) \).
  3. If \( \beta \subseteq result \), then \( \alpha \rightarrow \beta \) is preserved.

If all FDs in \( F \) are preserved, the decomposition is dependency preserving.

Worked Example 1

\( R(A,B,C,D) \), \( F = \{ A \rightarrow B, B \rightarrow C, C \rightarrow D \} \). Decompose into \( R_1(A,B) \), \( R_2(B,C) \), \( R_3(C,D) \). Test \( A \rightarrow D \).

  • Start result = {A}.
  • Iteration 1:
    • For \( R_1 \): \( T = \{A\} \cap \{A,B\} = \{A\} \). Compute \( T^+ = \{A,B\} \) (using \( A \rightarrow B \)). Intersection with \( R_1 \) = \{A,B\}. Add to result → result = \{A,B\}.
    • For \( R_2 \): \( T = \{A,B\} \cap \{B,C\} = \{B\} \). \( T^+ = \{B,C\} \) (using \( B \rightarrow C \)). Intersection with \( R_2 \) = \{B,C\}. Add → result = \{A,B,C\}.
    • For \( R_3 \): \( T = \{A,B,C\} \cap \{C,D\} = \{C\} \). \( T^+ = \{C,D\} \). Intersection = \{C,D\}. Add → result = \{A,B,C,D\}.
  • Now result contains D, so \( A \rightarrow D \) is preserved.

Worked Example 2 (Not preserved)

\( R(A,B,C) \), \( F = \{ AB \rightarrow C, C \rightarrow A \} \). Decompose into \( R_1(A,C) \), \( R_2(B,C) \). Test \( AB \rightarrow C \).

  • Start result = {A,B}.
  • Iteration 1:
    • \( R_1 \): \( T = \{A,B\} \cap \{A,C\} = \{A\} \). \( A^+ \) using F: from \( C \rightarrow A \) we don't get C from A alone; no FD with left side A. So \( T^+ = \{A\} \). Intersection = \{A\}. No change.
    • \( R_2 \): \( T = \{A,B\} \cap \{B,C\} = \{B\} \). \( B^+ = \{B\} \). No change.
  • Result remains {A,B}. Does not contain C. So \( AB \rightarrow C \) is not preserved.

Common Pitfalls

  • Forgetting to compute \( T^+ \) using the original \( F \), not just the projected FDs.
  • Not iterating until convergence; the algorithm may need multiple passes.

Practice Hints

This algorithm is polynomial and efficient. Use it for exam problems. Always write down the intermediate result sets.

Tutorial 5.2: Canonical Cover (Recap)

Algorithm / Procedure

  1. Union: Replace any set of FDs \( \alpha \rightarrow \beta_1, \alpha \rightarrow \beta_2, \dots, \alpha \rightarrow \beta_k \) with \( \alpha \rightarrow \beta_1\beta_2\dots\beta_k \).
  2. Find extraneous attributes:
    • For each FD \( \alpha \rightarrow \beta \), for each attribute \( A \in \alpha \), test if \( A \) is extraneous on left: compute \( (\alpha - \{A\})^+ \) using the current set of FDs. If it contains \( \beta \), remove \( A \) from \( \alpha \).
    • For each FD \( \alpha \rightarrow \beta \), for each attribute \( B \in \beta \), test if \( B \) is extraneous on right: compute \( \alpha^+ \) using the set where \( \alpha \rightarrow \beta \) is replaced by \( \alpha \rightarrow (\beta - \{B\}) \). If it contains \( B \), remove \( B \) from \( \beta \).
  3. Repeat until no changes.

Worked Example

\( F = \{ A \rightarrow BC, AB \rightarrow C, A \rightarrow D, D \rightarrow C \} \).

  • Union: already left sides unique? \( A \rightarrow BC \) and \( A \rightarrow D \) can be combined: \( A \rightarrow BCD \). So set becomes \( \{ A \rightarrow BCD, AB \rightarrow C, D \rightarrow C \} \).
  • Check \( AB \rightarrow C \): Is \( A \) extraneous? Compute \( B^+ \) using F: \( B^+ = \{B\} \) (no FD with left B). So not extraneous. Is \( B \) extraneous? Compute \( A^+ \) using F: \( A^+ = \{A,B,C,D\} \) (since \( A \rightarrow BCD \) and \( D \rightarrow C \)). Does it contain \( C \)? Yes. So \( B \) is extraneous. Remove B: becomes \( A \rightarrow C \). But \( A \rightarrow C \) is already implied by \( A \rightarrow BCD \). So \( AB \rightarrow C \) is redundant and can be removed entirely.
  • Now set = \( \{ A \rightarrow BCD, D \rightarrow C \} \).
  • Check \( A \rightarrow BCD \) for extraneous on right: Is \( C \) extraneous? Compute \( A^+ \) using \( A \rightarrow BD \) (if we remove C) and \( D \rightarrow C \)? Actually we need to test: with \( F' = \{ A \rightarrow BD, D \rightarrow C \} \), compute \( A^+ = \{A,B,D\} \) then using \( D \rightarrow C \), add C → \{A,B,C,D\}. So C is not extraneous because without \( A \rightarrow C \) directly, we still get C via D. But careful: The test for right extraneous: Remove C from RHS of \( A \rightarrow BCD \), get \( A \rightarrow BD \). Then compute \( A^+ \) using the new set (which includes \( A \rightarrow BD \) and \( D \rightarrow C \)). \( A^+ = \{A,B,D\} \) then using \( D \rightarrow C \) gives C, so \( A^+ \) contains C, meaning that \( A \rightarrow C \) is implied, so C is extraneous. Yes, we can remove C. So FD becomes \( A \rightarrow BD \).
  • Now set = \( \{ A \rightarrow BD, D \rightarrow C \} \).
  • Check \( A \rightarrow BD \): is D extraneous? Compute \( A^+ \) using \( A \rightarrow B \) only (remove D): \( A^+ = \{A,B\} \). Does not contain D. So not extraneous. is B extraneous? Compute \( A^+ \) using \( A \rightarrow D \): \( A^+ = \{A,D\} \) then \( D \rightarrow C \) gives C, but not B. So not extraneous. So canonical cover = \( \{ A \rightarrow BD, D \rightarrow C \} \).

Practice Hints

Canonical cover is not unique; different elimination orders may give different but equivalent covers. Always re‑apply union after each removal.
