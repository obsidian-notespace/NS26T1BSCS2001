10. Derived Rules from Armstrong’s Axioms

Definition

Derived rules are additional inference rules that follow from Armstrong’s three axioms. They are not axioms themselves but theorems that simplify FD manipulation.

Explanation

The three most useful derived rules are:

  • Union (or Additivity): If \( \alpha \rightarrow \beta \) and \( \alpha \rightarrow \gamma \), then \( \alpha \rightarrow \beta\gamma \).
    Proof: Augment \( \alpha \rightarrow \beta \) with \( \alpha \) to get \( \alpha\alpha \rightarrow \alpha\beta \) i.e., \( \alpha \rightarrow \alpha\beta \). Then augment \( \alpha \rightarrow \gamma \) with \( \beta \) to get \( \alpha\beta \rightarrow \beta\gamma \). By transitivity, \( \alpha \rightarrow \beta\gamma \).
  • Decomposition (or Projectivity): If \( \alpha \rightarrow \beta\gamma \), then \( \alpha \rightarrow \beta \) and \( \alpha \rightarrow \gamma \).
    Proof: Since \( \beta \subseteq \beta\gamma \), reflexivity gives \( \beta\gamma \rightarrow \beta \). Transitivity with \( \alpha \rightarrow \beta\gamma \) gives \( \alpha \rightarrow \beta \). Similarly for \( \gamma \).
  • Pseudo‑transitivity: If \( \alpha \rightarrow \beta \) and \( \beta\gamma \rightarrow \delta \), then \( \alpha\gamma \rightarrow \delta \).
    Proof: Augment \( \alpha \rightarrow \beta \) with \( \gamma \) to get \( \alpha\gamma \rightarrow \beta\gamma \). Then transitivity with \( \beta\gamma \rightarrow \delta \) gives \( \alpha\gamma \rightarrow \delta \).

These rules are used to simplify FD sets and to compute attribute closures efficiently.

Intuition / Mental Model

If you know that A determines B and A determines C, then obviously A determines both B and C together. Conversely, if A determines B and C together, it certainly determines B alone. That’s the union and decomposition rules.

Key Definitions

  • Union rule: Combine right sides of FDs with same left side.
  • Decomposition rule: Split right side into individual attributes.
  • Pseudo‑transitivity: Allows “extra” attributes on the left of the second FD.

Common Mistakes

  • Misconception: “Union rule can be applied to any two FDs.” (Correction: The left sides must be identical.)
  • Misconception: “Decomposition can split left side.” (Correction: Only the right side can be decomposed; left side splitting is not generally valid.)

Key Takeaways

• Union: combine same left side.
• Decomposition: split right side.
• Pseudo‑transitivity: chain with extra attributes.
• These are shortcuts, not new axioms.

11. Closure of a Set of Attributes (\( \alpha^+ \))

Definition

Given a set of functional dependencies \( F \) and a set of attributes \( \alpha \), the closure of \( \alpha \) (denoted \( \alpha^+ \)) is the set of all attributes that are functionally determined by \( \alpha \) under \( F \).

Explanation

The closure algorithm is fundamental:

  1. Start with \( result = \alpha \).
  2. Repeatedly, for each FD \( \beta \rightarrow \gamma \) in \( F \), if \( \beta \subseteq result \), then add \( \gamma \) to \( result \).
  3. Continue until no more changes.

This algorithm runs in polynomial time and is extremely useful for:

  • Testing if \( \alpha \) is a superkey: \( \alpha^+ = \text{all attributes of R} \).
  • Testing if an FD \( \alpha \rightarrow \beta \) holds: check if \( \beta \subseteq \alpha^+ \).
  • Finding candidate keys: start with single attributes, compute closure; if not superkey, add attributes until it becomes superkey, then minimize.
  • Computing the canonical cover.

Intuition / Mental Model

Think of the closure algorithm as a “snowball” that starts with the given attributes and rolls through the FDs, picking up all attributes that are forced to be determined.

Key Definitions

  • Closure: \( \alpha^+ \) = set of all attributes determined by \( \alpha \).
  • Superkey test: \( \alpha^+ = R \) (all attributes).
  • Candidate key: minimal superkey.

Formulas / Example

Given \( F = \{ A \rightarrow B, B \rightarrow C, BC \rightarrow D \} \), compute \( \{A\}^+ \).

  1. Start: result = {A}.
  2. Use \( A \rightarrow B \): add B → {A, B}.
  3. Use \( B \rightarrow C \): add C → {A, B, C}.
  4. Use \( BC \rightarrow D \): BC ⊆ result, so add D → {A, B, C, D}.
  5. No more FDs apply. Result = {A, B, C, D}.

Thus \( A^+ = \{A,B,C,D\} \).

Common Mistakes

  • Misconception: “The closure algorithm must consider FDs in a fixed order.” (Correction: Order doesn’t matter; you may need to iterate until no change.)
  • Misconception: “If an FD’s left side is not exactly in result, you cannot use it.” (Correction: If a subset of the left side is in result, you cannot use it; the entire left side must be a subset of result.)

Edge Cases

  • If \( F \) is empty, \( \alpha^+ = \alpha \).
  • If \( \alpha \) already includes all attributes, closure stops immediately.

Real‑world Application

Used in database design tools to automatically find all candidate keys of a relation from given FDs.

Key Takeaways

• Closure algorithm: start with \( \alpha \), repeatedly add right sides of FDs whose left side is already in result.
• \( \alpha^+ = R \) ⇔ α is a superkey.
• \( \beta \subseteq \alpha^+ \) ⇔ \( \alpha \rightarrow \beta \) holds.
• Candidate keys are minimal superkeys.
• The algorithm is polynomial and efficient.

12. Boyce‑Codd Normal Form (BCNF)

Definition

A relation schema \( R \) is in Boyce‑Codd Normal Form (BCNF) with respect to a set of FDs \( F \) if for every FD \( \alpha \rightarrow \beta \) in \( F^+ \) (the closure) that is non‑trivial, \( \alpha \) is a superkey of \( R \).

Explanation

BCNF is a stricter normal form than 3NF. It eliminates all redundancy caused by non‑key FDs. In other words, every non‑trivial FD must have a superkey on the left side.

For example, consider \( R(A,B,C) \) with \( A \rightarrow B \) and \( B \rightarrow C \), and candidate key \( A \). The FD \( B \rightarrow C \) violates BCNF because \( B \) is not a superkey. To fix, we decompose into \( R1(B,C) \) and \( R2(A,B) \).

BCNF decomposition is always lossless, but it may not preserve all functional dependencies. This is a known trade‑off.

Intuition / Mental Model

BCNF says: every arrow in the FD diagram must start from a superkey. If there’s an arrow starting from a non‑key, that’s a violation.

Key Definitions

  • BCNF: Every non‑trivial FD has a superkey left side.
  • BCNF violation: A non‑trivial FD \( \alpha \rightarrow \beta \) where \( \alpha \) is not a superkey.

Decomposition for BCNF

If \( \alpha \rightarrow \beta \) violates BCNF, decompose \( R \) into: \[ R_1 = \alpha \cup \beta, \quad R_2 = R - (\beta - \alpha) \] Then recursively decompose each if needed.

Common Mistakes

  • Misconception: “BCNF eliminates all redundancy.” (Correction: BCNF eliminates redundancy caused by FDs, but other types (e.g., multi‑valued dependencies) may still cause redundancy.)
  • Misconception: “BCNF decomposition always preserves FDs.” (Correction: Not true; 3NF is used when dependency preservation is required.)

Edge Cases

  • A relation with only one candidate key and all FDs having that key on left side is automatically in BCNF.
  • If there are multiple overlapping candidate keys, BCNF may be harder to achieve.

Real‑world Application

Most OLTP (online transaction processing) systems aim for BCNF to avoid update anomalies, even if it means some loss of dependency preservation.

Key Takeaways

• BCNF requires every non‑trivial FD to have a superkey left side.
• BCNF decomposition is lossless but may not preserve dependencies.
• Violating FDs are decomposed using \( \alpha\beta \) and \( R - (\beta-\alpha) \).
• BCNF is stronger than 3NF.

13. Third Normal Form (3NF)

Definition

A relation schema \( R \) is in Third Normal Form (3NF) if for every FD \( \alpha \rightarrow \beta \) in \( F^+ \), at least one of the following holds:

  1. \( \alpha \rightarrow \beta \) is trivial, or
  2. \( \alpha \) is a superkey of \( R \), or
  3. Each attribute in \( \beta - \alpha \) is a prime attribute (i.e., part of some candidate key).

Explanation

3NF relaxes BCNF slightly: it allows a non‑key FD if the right side consists entirely of prime attributes (attributes that belong to some candidate key). This relaxation ensures that we can always achieve a lossless and dependency‑preserving decomposition.

Example: \( R(A,B,C) \) with \( A \rightarrow B \) and \( B \rightarrow C \), candidate keys \( A \) and \( B \). The FD \( B \rightarrow C \) is non‑trivial and \( B \) is a superkey? Actually, \( B \) is a candidate key (since \( B \rightarrow C \) and \( A \rightarrow B \), so \( B \) determines all attributes). Wait, need careful: If candidate keys are A and B, then B is a superkey, so it’s actually in BCNF. A better example: \( R(A,B,C,D) \) with candidate keys \( AB \) and \( BC \). FD \( A \rightarrow D \) violates BCNF because A is not a superkey. But if D is prime? No, D is not in any candidate key. So not allowed in 3NF either. The classic 3NF‑but‑not‑BCNF example is with two overlapping candidate keys, e.g., \( R(A,B,C) \) with \( AB \rightarrow C \) and \( C \rightarrow B \). Candidate keys: \( AB \) and \( AC \). FD \( C \rightarrow B \) is non‑trivial, left side C is not a superkey (C alone does not determine A), but right side B is prime (part of candidate key AB). So it satisfies 3NF but not BCNF.

Intuition / Mental Model

3NF says: if a non‑superkey determines something, that something must be part of some key. This allows some controlled redundancy but guarantees dependency preservation.

Key Definitions

  • Prime attribute: An attribute that appears in at least one candidate key.
  • Non‑prime attribute: Not in any candidate key.

Comparison with BCNF

Property BCNF 3NF
Lossless decomposition Always Always
Dependency preservation Not always Always possible
Redundancy elimination Maximum Less than BCNF

Common Mistakes

  • Misconception: “3NF is always enough for practical databases.” (Correction: BCNF is often preferred; 3NF is used when dependency preservation is critical.)
  • Misconception: “Prime attributes cannot be determined by non‑keys.” (Correction: 3NF allows that, BCNF does not.)

Key Takeaways

• 3NF allows a non‑key FD if the right side is prime.
• Every BCNF relation is also in 3NF (but not vice versa).
• 3NF decomposition can always be lossless and dependency‑preserving.
• Use 3NF when you must preserve all FDs; use BCNF for maximum anomaly elimination.
